TASK 1.2
New table — siteCoils
convex/schema.ts

The transmit coil is Gate 6 of the matrix. Clinics must register which coils they have, what type each is, and which scanners they're compatible with. This table is the Site_Coils sheet from Tom's site registry.

siteCoils: defineTable({
  siteId:              v.id('clinics'),            // clinic this coil belongs to
  coilDisplayName:     v.string(),                // vendor label, e.g. "Body 18 surface coil"
  coilType:            v.string(),                // controlled: transmit_coil_type vocab
  fieldStrength:       v.string(),                // '1.5T' | '3T' | '7T' | '0.55T'
  compatibleScannerIds: v.array(v.id('scanners')), // all scanners this coil works with at this site
  txCapable:           v.boolean(),               // transmit-capable = resolver uses this
  rxCapable:           v.boolean(),               // receive-capable
  channelCount:        v.optional(v.number()),
  manufacturer:        v.optional(v.string()),
  modelNumber:         v.optional(v.string()),
  status:              v.union(v.literal('active'), v.literal('retired')),
  recordState:         v.union(v.literal('Unconfirmed'), v.literal('Confirmed')),
  entryDate:           v.optional(v.string()),
  notes:               v.optional(v.string()),
})
.index('by_site', ['siteId'])
.index('by_site_status', ['siteId', 'status'])
⚠️
S-3 calibration rule (from Tom's site registry): every scanner in compatibleScannerIds must have a matching field strength. Cross-field-strength associations should be blocked in the mutation handler.

TASK 1.5
New table — riskBenefitRecords (pre-scan decision gate)
convex/schema.ts

Tom's ruling P-2: the Risk-Benefit decision is separate from the scan event. This is the pre-scan gate — clinician + radiologist sign off on proceeding. Contains the matrix resolution snapshot at the time of the decision.

riskBenefitRecords: defineTable({
  patientId:            v.id('patients'),
  clinicId:             v.id('clinics'),
  staffId:              v.id('staff'),

  // Clinical decision
  indication:           v.string(),
  decision:             v.union(v.literal('Proceed'), v.literal('Do Not Proceed'), v.literal('Deferred')),
  reasonForProceeding:  v.optional(v.string()),

  // Hardware planned
  scannerId:            v.optional(v.id('scanners')),
  coilId:               v.optional(v.id('siteCoils')),
  bodyRegion:           v.optional(v.string()),

  // Matrix output snapshot (captured at time of decision — never re-resolved)
  resolvedOutcomeCache:    v.optional(v.string()),    // 'PASS' | 'FAIL' | 'UNRESOLVED'
  resolvedConditionIds:    v.optional(v.array(v.string())),
  resolvedConstraintsJson: v.optional(v.string()),    // JSON constraint set
  resolutionTimestamp:     v.optional(v.number()),

  // Sign-off
  clinicianName:        v.string(),
  clinicianRole:        v.string(),
  clinicianSignedAt:    v.number(),
  radiologistName:      v.optional(v.string()),
  radiologistSignedAt:  v.optional(v.number()),

  // Document link
  signedFormStorageId:  v.optional(v.id('_storage')),

  linkedScanEventId:    v.optional(v.id('scanEvents')),

TASK 1.6
New table — scanEvents (post-scan audit log)
convex/schema.ts

Tom's ruling P-3: the scan event log stores a frozen resolution snapshot — the exact condition IDs resolved at the time of scanning, the parameters used, and the outcome. This snapshot is never re-read for future resolutions; it exists only for medico-legal audit.

scanEvents: defineTable({
  patientId:              v.id('patients'),
  clinicId:               v.id('clinics'),
  staffId:                v.id('staff'),
  scannerId:              v.id('scanners'),
  coilId:                 v.optional(v.id('siteCoils')),
  bodyRegion:             v.string(),
  rvbRecordId:            v.optional(v.id('riskBenefitRecords')),

  // Actual parameters used at console
  fieldStrengthUsed:      v.string(),
  operatingModeUsed:      v.optional(v.string()),
  sarWbActual:            v.optional(v.number()),    // W/kg actually applied
  sarHeadActual:          v.optional(v.number()),
  b1RmsActual:            v.optional(v.number()),    // µT
  slewRateActual:         v.optional(v.number()),    // T/m/s
  scanTimeMins:           v.optional(v.number()),

  // Outcome
  outcome:                v.union(v.literal('Completed'), v.literal('Early Termination'), v.literal('Aborted'), v.literal('Not Performed')),
  earlyTerminationReason: v.optional(v.string()),
  adverseEvents:          v.optional(v.string()),
  postScanDeviceCheck:    v.optional(v.string()),
  patientPostScanNotes:   v.optional(v.string()),

  // AUDIT SNAPSHOT — P-3 ruling. Frozen at scan time. Never re-read for future resolutions.
  resolvedOutcomeCache:   v.string(),                // 'PASS' | 'FAIL' | 'UNRESOLVED'
  resolvedConditionIds:   v.array(v.string()),       // exact condition IDs used
  resolvedConstraintsJson: v.string(),               // full constraint set as JSON

  createdAt:              v.number(),

.index('by_patient', ['patientId'])
.index('by_clinic', ['clinicId'])

TASK 2.4
New file convex/matrix.ts — the gate-walk resolver (core feature)
convex/matrix.ts

This is the primary new capability. A Convex action that implements the 10-gate walk from the prototype HTML, now running against real database rows. The prototype is the canonical spec — implementation follows it gate-by-gate.

💡
Key principle: closed-world. If no condition record exists for a given input combination, the resolver returns FAIL — never a guess, never "probably safe." Every gate either finds a matching record or stops.

Action signature:

resolveMatrix = action({
  args: {
    mode:                  v.union(v.literal('registry'), v.literal('manual')),
    // Registry mode
    patientId:             v.optional(v.id('patients')),
    // Manual mode
    manualDeviceId:        v.optional(v.id('devices')),
    manualLeadIds:         v.optional(v.array(v.id('devices'))),
    manualImplantLocation: v.optional(v.string()),
    manualIntegrityState:  v.optional(v.string()),
    // Hardware (both modes)
    scannerId:             v.id('scanners'),
    coilId:                v.optional(v.id('siteCoils')),
    bodyRegion:            v.string(),
  },
  handler: async (ctx, args) => { ... }
})

Return type:

{
  verdict:            'PASS' | 'FAIL' | 'UNRESOLVED',
  systems: [{
    label:            string,
    verdict:          'PASS' | 'FAIL' | 'UNRESOLVED',
    deviceName:       string,
    mriClassification: string,
    gates:            GateRow[],      // for display in the UI gate table
    constraints:      string[],       // what operator must set
    reasons:          string[],       // why verdict was reached
    resolvedConditionIds: string[],   // for the audit snapshot
  }],
  combinedConstraints: string[],      // deduped, most restrictive union
  weightAlert:        string | null,  // if patient weight > scanner table limit
}

Gate-walk implementation steps inside resolveSystem():

  • Gate 0 — fetch all conditions for this device via listConditionsByDevice. If none exist → FAIL (closed world — no record for any combination)
  • Gate 1 Eligibility — check patientDevice.deviceIntegrityState against condition leaves with eligibilityTier = 'Not Eligible'. If any prohibition leaf matches integrity state → FAIL (never overridable)
  • Gate 1b VNS special case — Group A leaves require confirmed upper-left-chest placement. Missing or different location → fail-closed to Group B. If no Group B leaves → UNRESOLVED
  • Gate 2 Field strength — filter candidates by fieldStrength matching scanner. Zero matches → FAIL
  • Gate 3 Spatial gradient — compare scanner.maxSpatialGradient vs condition's maxSpatialGradientCond. Exceeds → CHECK constraint added, continue
  • Gate 4 Slew rate — compare scanner.maxSlewRate vs condition's maxSlewRateCond. Exceeds → CHECK constraint added
  • Gate 5 Bore / orientation — compare scanner bore/orientation vs boreTypeRequired / fieldOrientationRequired. Mismatch → FAIL
  • Gate 6 Transmit coil — map coil's coilType through COIL_VOCAB_MAP, filter conditions by transmitCoilType. Zero matches → FAIL. Receive-only coil → resolved as body-transmit context.
  • Gate 6b Implant location — if remaining candidates are location-scoped, filter by patientDevice.implantLocation. Location missing / no match → UNRESOLVED
  • Gate 6c Lead model — if candidates are lead-scoped, filter by patient's lead device IDs. Unknown lead → UNRESOLVED
  • Gate 7 Operating mode — extract permitted modes from surviving candidates, add as constraint. No gate failure.
  • Gate 8 Isocentre / exclusion zone — map requested body region to isocentre position (superior/inferior C7). Select appropriate leaf. Check region against exclusionZoneDescription. Blocked region → FAIL
  • Gate 9 RF limit — extract B1+RMS or SAR from selected leaf. Per-lead limit with unknown lead → UNRESOLVED. Add as constraint.
  • Gate 10 Time / thermal — extract scan time, window, cooloff, temp monitoring. Add as constraints.

Multi-system handling (outer loop in resolve()):

  • Each system (grouped by systemGroupId) resolved independently
  • Combined verdict: any FAIL → combined is FAIL; any UNRESOLVED → UNRESOLVED; all PASS → PASS
  • Constraints: union of all system constraint sets (most restrictive numeric limits win per field)
  • Weight pre-check: if patient.weightKg > scanner.tableLimitKg, add a weight alert before the gate walk runs
🔴
MR Unsafe is never overridable. Any device with mriClassification = 'MR Unsafe' stops the resolver immediately with FAIL, regardless of any other conditions. Do not even attempt the gate walk.

7
Cross-cutting Quality Frontend Functions
Platform-wide consistency, safety principles, and edge case handling. These can be done incrementally alongside Phases 4–6.
TASK 7.1
Closed-world enforcement on device pages

Any device display that shows MRI information must check whether condition records exist. If a device has mriClassification = 'MR Conditional' but zero condition rows in deviceConditions, it must not show any MRI parameters — it shows instead:

"No condition records in database. Treat as unresolved — do not scan without consulting current manufacturer documentation."

This applies to: the admin device detail, the clinic device lookup, and the matrix resolver (Gate 0 failure).

TASK 7.2
mri_classification display standardisation

After Task 3.3 populates mriClassification everywhere, audit all places in the UI that display MRI status and update them to use mriClassification (ASTM standard) as the primary label. The legacy mriStatus field remains in the database for backward compatibility but should not be displayed to users.

MRI classification display rules:

  • MR Safe — green badge. "This device has been determined to present no known hazards in all MRI environments."
  • MR Conditional — amber badge. "Permitted under specific conditions — resolve against scanner and patient implant data before scanning."
  • MR Unsafe — red badge. "Do not scan. MR Unsafe devices present known hazards in MRI environments."
  • Classification Unknown — grey badge. "Classification not available — treat as MR Unsafe unless manufacturer documentation confirms otherwise."
TASK 7.3
Weight limit pre-check in the matrix

Before the gate walk runs, check if the selected scanner has tableLimitKg set and if the patient (registry mode) has weightKg set. If patient.weightKg > scanner.tableLimitKg, display a prominent CHECK warning:

"Patient weight [X kg] may exceed this scanner's rated table limit [Y kg]. Confirm with site engineering before positioning the patient."

This does not block the gate walk — it's a pre-check advisory, not a hard failure.

TASK 7.4
Recall warning on patient implant view

When viewing a patient's implant record, check the linked device's recalled flag. If true, display a prominent recall alert on the implant card:

"⚠️ RECALL NOTICE: [recallNotes]. Check with the patient's implanting surgeon before any scan or procedure."

Full serial-number-range matching (condition rows with serialNumberRange vs patient serial numbers) can be added as a follow-on task once the recall register is populated with range data.

TASK 7.5
Post-implant wait check in the matrix

If the selected device has postImplantWaitWeeks set, and the patient's implant record has implantDate, calculate whether the required wait period has elapsed. If not, display a CHECK warning:

"Post-implant wait period: [device] requires [N] weeks post-implant before MRI. Implanted [date] — [N] weeks from [date] = [date]. Confirm elapsed with surgeon."

Implementation notes All phases are designed to be non-destructive — no existing data is modified except by the explicit migration in Task 3.3. The matrix resolver (Task 2.4) is the most complex single task and should be built with reference to the prototype HTML as the canonical spec. Any ambiguity in the gate logic should default to the prototype's behaviour. Tom's schema documents are the authority for data structure decisions.

The plan is ordered for sequential execution: approve → Phase 1 (schema push to Convex) → Phase 2 (functions) → Phase 3 (seed data) → then Phases 4–7 can proceed in parallel across admin, clinic, and patient portals.
