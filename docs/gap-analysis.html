What the Matrix Prototype does (and what we need to replicate)
The gate-walk algorithm — runs through 10 sequential gates, stops at the first hard failure, outputs PASS / FAIL / UNRESOLVED
Gate 1
Eligibility / integrity
Is the system complete? Any broken/abandoned leads → hard FAIL, non-overridable. Patient's device_integrity_state is the join key.
Gate 2
Field strength
Does a condition record exist at this scanner's field strength? Absence = FAIL. Closed-world: no record = no permission.
Gate 3 & 4
Spatial gradient + slew rate
Scanner's max gradient (T/m) and slew rate (T/m/s) checked against the condition limit. Generates a CHECK constraint if scanner exceeds limit.
Gate 5
Bore type + orientation
Most conditions require closed-cylindrical, horizontal field. Open-bore or vertical → FAIL if condition doesn't match.
Gate 6
Transmit coil type
The coil is a gate, not a display field. Whole Body Transmit, T/R Head Volume, T/R Lower Extremity — each unlocks different leaves. Receive-only coils don't change body-transmit hazard.
Gate 6b & 6c
Generator location + lead model
Some conditions split by where the generator is implanted (chest vs abdomen). Different leads have different RF limits — lead model is required to select the right leaf.
Gate 7
Operating mode
Normal vs First Level — most active devices must be in a specific mode. Stored as a constraint in the output, not a gate that fails.
Gate 8
Exclusion zone / isocentre
Some devices prohibit scanning over certain body regions. Requested scan region checked against the condition's exclusion zone. VNS, for example, blocks chest/neck scans with body-coil.
Gate 9
RF limit (B1+RMS / SAR)
The specific RF energy limit for this leaf — whole-body SAR, head SAR, or B1+RMS in µT. Per-lead when lead model is known. Operator sets this at the console.
Gate 10
Time + thermal
Max scan time, scan window, cooloff period, patient temperature monitoring requirement. Stored as constraints in the output.
⚠️
Multi-system rule: when a patient has multiple implanted systems (e.g. pacemaker + DBS), every system must pass independently and the most restrictive combined constraint set governs. MR Unsafe is never overridable — one unsafe device = full stop regardless of everything else.
Gap 1 — Conditions data model Critical
Tom's schema has 46 columns per condition row. Our deviceConditions table has ~8 fields. This is the biggest single gap — the matrix cannot run without the full conditions structure.

Fields we're missing from each condition row:

🔴
device_integrity_state on the condition row is a hard gate. The condition row must declare what integrity state it applies to (Complete / Fractured-Suspected / Abandoned-Fragment). A patient with a broken lead hits a "Not Eligible" leaf — and the scan is refused. Without this field on the condition, the integrity check can't run.

Gap 3 — Scanner hardware data Partial
Our scanners table has manufacturer, model, field strength, bore type — but is missing the hardware specs the gate walk compares against conditions.
FieldStatusNotes
maxSlewRate Missing T/m/s — Gate 4 of the matrix. The condition sets a max slew rate limit; the scanner's rated max is compared. Siemens Skyra = 200 T/m/s, Free.Max = 100 T/m/s.
fieldOrientation Missing Horizontal vs Vertical — Gate 5. Vertical-field (stand-up) scanners are a different context; most device conditions only cover horizontal.
tableLimitKg Missing Patient weight suitability check — paired with patients.weightKg. Some scanners have 110 kg limits; bariatric scanners go higher. (Ruling S-5)
b1RmsMonitoringCapable Missing Some MR Conditional devices require B1+RMS monitoring at the console. If the scanner can't display it, the scan can't proceed for those devices.
maxSpatialGradient Exists Already in our schema — T/m. Gate 3 of the matrix. Good.
boreDiameter Exists Present, though Tom's schema stores this on the Scanner_Models reference layer (not the site instance).
🔴
No coils table at all. The transmit coil type is Gate 6 — arguably the most important hardware gate. A patient with a DBS device may scan at 3T with a head coil but not a body coil; with a lower extremity coil the B1+RMS limit is removed entirely. We have no coil data. We need a siteCoils table: coil display name, transmit_coil_type (controlled vocab), field strength, compatible scanner IDs, tx_capable flag.

Gap 5 — Clinical audit trail Missing
Two tables Tom considers essential before a scan event — the pre-scan decision gate and the post-scan record.
TableStatusWhat it does
Risk_Benefit_Registry Missing The pre-scan decision gate. Clinician + radiologist sign-off, clinical indication, reason for scanning despite risk, decision (Proceed / Do Not Proceed / Deferred), signed form reference. Medico-legally this is the document that says "we decided to scan."
Tom's ruling P-2: this is separate from the scan event log. RvB = decision; Scan Event = execution.
Scan_Event_Log Missing What actually happened: site, scanner used, parameters, outcome, adverse events, early termination, post-scan device check. Critically — the resolution snapshot: the exact condition IDs resolved on that day, the outcome, and a timestamp.
P-3 ruling: the snapshot is an audit record because the device DB moves underneath. A future resolution for the same patient might differ if device data is updated. Medico-legal requires capturing what was served at the time of the scan.

Concept gaps — how the resolver is supposed to think
These aren't data model issues — they're principles that the implementation needs to encode
Concept Closed-world principle
The resolver never assumes safety. If no condition record exists for a given combination (field strength + coil + region + integrity state), the answer is FAIL — not "probably fine." Absence of a record is never treated as safe. Our current device display doesn't enforce this — it shows what data we have, but doesn't say "no record = not permitted."
Concept Most restrictive governs
Patient with a pacemaker AND a DBS implant: both systems must pass their full gate walk independently. The combined constraint set is the union of all constraints from both systems. The most restrictive limits from each system apply simultaneously. The prototype implements this correctly.
Concept Audit snapshot (not live re-read)
When a scan is logged, the exact condition IDs resolved and the outcome are captured in the Scan_Event_Log alongside a timestamp. Future reads of that event show what was true on the day — not what the device DB says today. The device DB may be updated (new conditions, corrections) but the historical scan record is frozen. This is medico-legal.
Concept Free-text never resolves
If a patient has an implant recorded as free text only (no device_id link to a governed record), the matrix outputs UNRESOLVED — not a guess. The prototype shows this clearly: "Implant recorded as free text only — link to a governed device record before this system can be resolved." We need to enforce this in the UI.

Suggested build order — when you're ready to start
This is a read-only review — not implementation yet. But here's the logical sequence when we do move.
1
Uplift conditions model (46-column structure)
Expand deviceConditions to include the full gate-walk fields: eligibility_tier, device_integrity_state, transmit_coil_type, operating_mode, bore_type_required, field_orientation_required, max_sar_head, max_slew_rate, max_spatial_gradient, max_scan_time_mins, cooloff_period_mins, exclusion_zone_applies, exclusion_zone_description, isocentre_restriction, region_excluded, patient_preconditions. This is the database foundation — nothing else can run without it.
2
Add device_integrity_state to patientDevices + implant_location as controlled vocab
Gate 1 and Gate 6b of the matrix both need these. device_integrity_state: Complete / Fractured-Suspected / Abandoned-Fragment / Explanted-Partial / Not Stated. implant_location: Chest / Upper Left Chest / Abdomen / Cranial / Left vagus nerve / etc. — controlled list, not free text.
3
Add siteCoils table + scanner hardware fields
The coil gate is Gate 6 — the most nuanced gate. Add siteCoils with transmit_coil_type (controlled vocab), txCapable, compatibleScannerIds. Add to scanners: maxSlewRate, fieldOrientation, tableLimitKg, b1RmsMonitoringCapable.
4
Build the matrix resolver (gate-walk function)
A Convex query / action that takes patient + site + scanner + coil + body region and runs the 10-gate walk, returning PASS / FAIL / UNRESOLVED with the constraint set. The prototype HTML is the spec — it's a beautiful reference implementation. The logic is well-defined; the hard part is the data, which steps 1–3 address.
5
Standardise mri_classification vocabulary platform-wide
Make mriClassification (MR Safe / MR Conditional / MR Unsafe / Classification Unknown) mandatory and the primary display field. Deprecate the legacy mriStatus field or derive it from mriClassification.
6
Risk-Benefit Registry + Scan Event Log
These two tables are the clinical audit trail. The matrix resolver output feeds into the Risk-Benefit record (clinician decides to proceed based on the resolved constraints). The Scan Event Log captures what actually happened and snapshots the exact condition IDs used — medico-legally required.
7
Device category differentiation + passive implant data model
Passive implants (hip replacements, stents, screws) need very different fields (deflection angle, torque, artifact extent, material ferromagnetic). Active cardiac devices need chamber count, programming requirements. Can be tackled after the matrix is running for active devices.
Important: this is assistive review only. Tom's schema is the governed authority. Any implementation decisions — particularly around clinical safety logic, the verification_status vocabulary, and the conditions data structure — should be confirmed with Tom before coding. The gate-walk logic in the matrix prototype is the reference implementation for the resolver. Where the prototype and this analysis disagree, trust the prototype code.
